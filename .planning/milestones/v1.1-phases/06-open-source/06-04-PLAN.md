<task type="auto">
  <name>Task 3: Build fresh APK from v1.1 tag and create GitHub Release</name>
  <files></files>
  <read_first>
    - .planning/phases/06-open-source/06-RESEARCH.md ¬ßPattern 2 (release creation) and ¬ßCommon Pitfalls 7 (APK must exist before release)
    - CHANGELOG.md (used as release notes body)
  </read_first>
  <action>
    Execute this EXACT sequence to guarantee the uploaded APK matches the v1.1-tagged source:

    **Step A ‚Äî Remember current branch/position:**
    ```bash
    ORIGINAL_BRANCH=$(git branch --show-current)
    # Expect: main
    ```

    **Step B ‚Äî Checkout the v1.1 tag (detached HEAD ‚Äî intentional):**
    ```bash
    git checkout v1.1
    git log -1 --oneline
    # Capture SHA for SUMMARY
    ```

    **Step C ‚Äî Clean build from the tagged commit:**
    ```bash
    ./gradlew clean
    ./gradlew assembleDebug
    ```
    Confirm APK exists:
    ```bash
    ls -la app/build/outputs/apk/debug/app-debug.apk
    # Capture file size + sha256 for SUMMARY:
    sha256sum app/build/outputs/apk/debug/app-debug.apk
    ```

    **Step D ‚Äî Create the GitHub Release with the APK attached:**
    ```bash
    gh release create v1.1 \
      app/build/outputs/apk/debug/app-debug.apk \
      --title "v1.1 ‚Äî Initial Public Release" \
      --notes-file CHANGELOG.md
    ```

    **Step E ‚Äî Return to main branch:**
    ```bash
    git checkout "$ORIGINAL_BRANCH"
    # Expect: main
    ```
    (The build artifacts in `app/build/` are gitignored, so returning doesn't modify any tracked files.)

    **Step F ‚Äî Verify release:**
    ```bash
    gh release view v1.1 --json tagName,name,assets
    # MUST contain assets entry with name "app-debug.apk"

    gh release download v1.1 --pattern "app-debug.apk" --dir /tmp/release-verify
    # Re-download and compare size/sha with the local build
    sha256sum /tmp/release-verify/app-debug.apk
    ```

    DO NOT:
    - Build from `main` without checking out the tag ‚Äî APK binary could drift from what the tag's source would produce.
    - Use `softprops/action-gh-release@v3` in CI as a substitute ‚Äî that's a future enhancement, not v1.1 scope.
    - Use `--draft` ‚Äî release should be immediately visible.
    - Use `--prerelease` ‚Äî v1.1 is a real release.
    - Forget to `checkout $ORIGINAL_BRANCH` at the end ‚Äî leaving the repo detached is confusing for the user.
  </action>
  <verify>
    <automated>gh release view v1.1 --json assets -q '.assets[].name' | grep -q "app-debug.apk" && [ "$(git branch --show-current)" = "main" ]</automated>
  </verify>
  <acceptance_criteria>
    - `gh release view v1.1 --json tagName -q .tagName` returns `v1.1`.
    - `gh release view v1.1 --json assets -q '.assets[].name'` contains `app-debug.apk`.
    - Re-downloaded APK sha256 matches local build sha256 (proves no corruption in upload).
    - `git branch --show-current` returns `main` (we're back on the branch, not detached).
  </acceptance_criteria>
  <done>Release v1.1 is live on GitHub with an APK sideload users can download immediately. APK is reproducibly built from the v1.1-tagged commit.</done>
